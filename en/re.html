Requirement Engineering

Defining what you want

A prompt is functionally a requirements document. Requirement Engineering teaches us to articulate clearly what a system should do. The same techniques help us formulate better prompts: What should happen? What should not happen? What's the input? What's the output?

Theory Exercises Deep Dive Resources

Theory

Key Points

  • A prompt is functionally a requirements document
  • User Stories are more accessible than formal specifications
  • LLMs guess when requirements are vague – with unpredictable results
  • INVEST criteria help formulate clear requirements
  • Core questions: What to do? What not to do? Input? Output? Success criteria?

Concepts

User Story
"As a [role] I want [function] so that [benefit]" – lightweight requirements format
INVEST Criteria
Independent, Negotiable, Valuable, Estimable, Small, Testable – criteria for good requirements
Acceptance Criteria
Concrete conditions that must be met for a requirement to be fulfilled
Scope
Clear definition of what is included in the project and what is not
Edge Cases
Special cases and exceptions that must be considered

From User Story to Prompt

user_story_example.txt 
# User Story:
"As an archivist I want to batch rename images
so that they follow our naming convention."

# Translated to Prompt:
"""
Task: Python script for batch renaming images

Input: Folder with JPG files (e.g., IMG_001.jpg)
Output: Renamed files (e.g., 2024_exhibition_001.jpg)

Requirements:
- Prefix from command line parameter
- Keep sequential numbering
- Preserve original extension
- Error if file already exists

Not part of scope:
- No subfolders
- No other formats than JPG
"""

Exercises

RE-1: User Stories for AI Tasks

Formulate requirements in a structured format.

Learning Objectives:
  • Use User Story format
  • Identify acceptance criteria
  • Distinguish in-scope from out-of-scope
template.txt 
# User Story Template:
As a [role]
I want [function]
so that [benefit]

# Acceptance Criteria:
- [ ] Criterion 1
- [ ] Criterion 2
- [ ] Criterion 3

# In Scope:
- ...

# Out of Scope:
- ...

Task: Formulate 3 User Stories for tasks in your work area. Define at least 3 acceptance criteria for each.

Reflection:
  • Where was delimitation difficult?
  • Which criteria are difficult to test?
  • How would you translate the User Story to a prompt?

RE-2: INVEST Check

Evaluate requirements using INVEST criteria.

Learning Objectives:
  • Apply INVEST criteria
  • Identify weak requirements
  • Improve requirements iteratively
invest_check.txt 
# INVEST Criteria:
I - Independent: Can it stand on its own?
N - Negotiable: Are details flexible?
V - Valuable: What is the benefit?
E - Estimable: Can effort be estimated?
S - Small: Achievable in one session?
T - Testable: When is it "done"?

# Example Check:
"Analyze all data and create a report"
- Independent: ✓
- Negotiable: ? (what exactly?)
- Valuable: ✓
- Estimable: ✗ (too vague)
- Small: ✗ (too large)
- Testable: ✗ (when is it "good"?)

Task: Check your User Stories from RE-1 against INVEST criteria. Improve the weakest points.

Deep Dive

The Five Core Questions

Before writing a prompt, answer these questions:

  1. What to do? – The main task
  2. What not to do? – Explicit exclusions
  3. What's the input? – Data format, source, size
  4. What's the output? – Format, location, naming
  5. What is success? – Testable criteria

LLMs and Vague Requirements

When requirements are vague, LLMs:

  • Make assumptions that may not match your intent
  • Choose solutions that seem "typical" but may be inappropriate
  • Produce inconsistent results across multiple attempts
  • Cannot ask clarifying questions proactively

Consequence: Spend time upfront on clear requirements rather than debugging later.

From Requirements to Prompts

User Story → Task Description
The "I want" part becomes the main instruction
Acceptance Criteria → Constraints
List as explicit requirements
Out of Scope → Negative Constraints
"Do NOT include...", "Ignore..."
Edge Cases → Examples
Show how to handle special cases
