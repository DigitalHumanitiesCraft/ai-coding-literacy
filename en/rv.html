RV

Review

Critically evaluating code

Generated code looks correct on the surface – but appearances are deceptive. LLMs produce syntactically correct but semantically wrong code, hallucinated libraries, and subtle security vulnerabilities. Systematic review is not optional.

Theory Exercises Deep Dive Resources

Theory

Key Points

  • EvalPlus: Pass rates drop 19-28.9% with more thorough testing
  • Tambon et al.: 10 bug patterns in LLM-generated code
  • 29.5% of Python snippets from Copilot have security vulnerabilities
  • Hallucinated libraries: Functions/packages that don't exist
  • Intent Conflicting: Code contradicts requirements
  • Context Inconsistency: Code contradicts provided context

Concepts

Hallucinated Objects
Non-existent functions, libraries, or APIs that LLMs confidently generate
Intent Conflicting
Code contradicts the stated requirement – does something else
Context Inconsistency
Code contradicts information provided in the context
Metamorphic Testing
Paraphrased prompts to check consistency of outputs
Bug Patterns
Recurring types of errors in LLM-generated code

Common Bug Patterns

Hallucination
  • Non-existent library functions
  • Wrong API signatures
  • Made-up package names
Logic Errors
  • Off-by-one errors
  • Wrong boundary conditions
  • Incorrect edge case handling
Security Issues
  • SQL injection vulnerabilities
  • Missing input validation
  • Insecure defaults

Exercises

RV-1: Bug Hunting

Find errors in generated code systematically.

Learning Objectives:
  • Identify common bug patterns
  • Test edge cases
  • Verify library imports
buggy_example.py 
# LLM-generated code to find duplicate files
import hashlib
from pathlib import Path

def find_duplicates(folder):
    hashes = {}
    duplicates = []

    for file in Path(folder).glob('*'):
        hash = hashlib.md5(file.read_bytes()).hexdigest()
        if hash in hashes:
            duplicates.append((hashes[hash], file))
        hashes[hash] = file

    return duplicates

# What are the problems?
# 1. Doesn't handle directories (will crash)
# 2. Doesn't handle unreadable files
# 3. Only checks immediate children, not subdirs
# 4. Variable 'hash' shadows builtin

Task: Have an LLM generate code for a task in your domain. Systematically check: Does it run? Edge cases? Error handling? Security?

Reflection:
  • What bug patterns did you find?
  • Which errors were obvious, which subtle?
  • How would you improve your review process?

RV-2: Metamorphic Testing

Check consistency with paraphrased prompts.

Learning Objectives:
  • Apply metamorphic testing
  • Identify inconsistencies
  • Evaluate robustness
metamorphic_test.txt 
# Original Prompt:
"Write a function that sorts a list of dates"

# Paraphrased Versions:
"Create a function to order dates chronologically"
"Implement date sorting for a list"
"Write code to arrange dates from earliest to latest"

# Compare:
- Do all versions handle the same date formats?
- Same edge cases (empty list, single item)?
- Same error handling?
- Same output format?

Task: Create 3 paraphrased versions of a prompt. Compare the outputs. Document differences.

Reflection:
  • How consistent were the outputs?
  • Which version produced the best result?
  • What does this tell you about prompt sensitivity?

Deep Dive

The Review Checklist

  1. Does it run? – Syntax errors, import errors
  2. Does it do what was asked? – Compare to requirements
  3. Edge cases? – Empty input, large input, special characters
  4. Error handling? – What happens when things go wrong
  5. Security? – Input validation, injection, sensitive data
  6. Dependencies? – Do libraries exist? Right versions?

Hallucination Categories

Library Hallucination
Package or function doesn't exist
API Hallucination
Function exists but with different signature
Behavioral Hallucination
Function exists but does something different
Version Hallucination
Feature exists in different version than assumed

Security Review Priorities

  • OWASP Top 10 – Common web vulnerabilities
  • Input validation – Never trust user input
  • Output encoding – Prevent injection
  • Authentication/Authorization – Access control
  • Sensitive data – Logging, storage, transmission
