Prompt Engineering

Developing effective prompts

Prompt Engineering refers to the development and optimization of input prompts. A prompt is more than a question – how you ask determines the type of answer. Schulhoff et al. (2024) document 58 different prompting techniques. For most applications, a few core strategies suffice.

Theory Exercises Deep Dive Resources

Theory

Key Points

  • Schulhoff et al. document 58 prompting techniques – for practice, a few suffice
  • Zero-Shot: Direct instruction without examples – for simple, unambiguous tasks
  • Few-Shot: 2-5 examples show format and handling of edge cases
  • Chain-of-Thought: "Think step by step" – for complex problems
  • Iterative refinement: Prompt → Output → Problem → Adjustment → repeat
  • Explicitly specify output format

Concepts

Zero-Shot
Direct instruction without examples – works for simple, unambiguous tasks
Few-Shot
2-5 examples in the prompt show the desired format and handling of edge cases
Chain-of-Thought
"Think step by step" – improves performance on complex, multi-step problems
Role Prompting
"You are an experienced..." – when specific expertise or perspective matters
Iteration
Prompt → Check output → Identify problem → Adjust prompt → repeat

When to Use Which Technique?

Zero-Shot – When the task is unambiguous:
  • "Translate this text to German"
  • "Format this data as JSON"
  • "Explain this code"
Few-Shot – When format matters:
  • Specific output formats
  • Demonstrating handling of edge cases
  • Domain-specific conventions
Chain-of-Thought – For complex logic:
  • Multi-step calculations
  • Logical reasoning
  • Debugging complex problems

Exercises

PE-1: Zero-Shot vs. Few-Shot

Understand the difference between direct instruction and example-based prompting.

Learning Objectives:
  • Distinguish Zero-Shot and Few-Shot
  • Recognize when examples help
  • Formulate examples effectively
prompting_strategies.txt 
# ZERO-SHOT:
"Extract the date from this text:
'The painting was acquired on March 15, 1823.'"

# FEW-SHOT:
"Extract the date from texts. Format as YYYY-MM-DD.

Text: 'Acquired in January 1820' -> 1820-01-00
Text: 'Dated to 3.5.1899' -> 1899-05-03
Text: 'The painting was acquired on March 15, 1823.' ->"

Task: Test both strategies. When does Few-Shot produce better results? When is Zero-Shot sufficient?

Reflection:
  • How did the outputs differ?
  • What edge cases did the examples cover?
  • When would you use Few-Shot in your work?

PE-2: Iterative Prompting

Step by step to the desired result through dialogue.

Learning Objectives:
  • Formulate feedback precisely
  • Know typical adjustments
  • Know when to start over
iteration.txt 
# Iteration 1:
"Write a script that resizes images."
# Problem: Unclear what size, format, folder

# Iteration 2:
"Write a Python script with Pillow.
Input: Folder path as argument
Task: Scale all .jpg to max 1200px width
Output: New files with suffix '_web'"
# Problem: Overwrites originals

# Iteration 3:
"... (as above)
Constraint: Don't overwrite original files
Constraint: Output to subfolder 'web/'"

Task: Start with a simple prompt for your project. Document 3 iterations: Problem → Adjustment → Result.

Reflection:
  • How many iterations did you need?
  • What type of feedback was most effective?
  • When would it have been better to start over?

Deep Dive

Chain-of-Thought in Practice

For complex problems, it helps to guide the LLM to think step by step:

chain_of_thought.txt 
# Without Chain-of-Thought:
"Calculate the total cost for 47 books at $23.50 with 19% tax."

# With Chain-of-Thought:
"Calculate the total cost for 47 books at $23.50 with 19% tax.
Think step by step:
1. Calculate the net price
2. Calculate the tax
3. Calculate the gross price
Show each step."

Typical Iteration Patterns

"Too general"
→ Add concrete examples
"Wrong format"
→ Explicitly specify output format or give an example
"Wrong assumptions"
→ Make constraints and boundaries explicit
"Too complex"
→ Break down problem, solve individually
"Inconsistent"
→ New prompt, start from scratch

Anti-Patterns to Avoid

  • Too many tasks – One prompt, one goal
  • Vague formulations – "good", "nice", "appropriate" → be specific
  • Contradictory instructions – "brief but comprehensive"
  • Excessive role-prompting – "You are the world's best expert" doesn't help
