Code Literacy

Reading and understanding code

Reading and writing code are separate competencies. You don't need to be a programmer to understand code – and understanding generated code is essential for using it safely. Even experts avoid deep understanding when possible; this is a feature, not a bug.

Theory Exercises Deep Dive Resources

Theory

Key Points

  • Reading and writing code are separate skills (Lopez et al.)
  • Skill hierarchy: Run → Trace → Explain → Write
  • Two models: Procedural ("what happens step by step?") and Functional ("what's the goal?")
  • Delocalized Plans: Code for one intention scattered across multiple locations
  • Even experts avoid deep understanding when possible
  • 39% of practitioners find LLM code more readable than human code

Concepts

Skill Hierarchy
Run → Trace → Explain → Write – each level requires the previous one
Procedural Model
"What happens step by step?" – Following program flow line by line
Functional Model
"What is the goal?" – Understanding the purpose without tracing every line
Delocalized Plans
Code belonging to one concept scattered across multiple files/functions
Code Comprehension
The ability to understand code at appropriate levels of abstraction

The Skill Hierarchy

Level 1: Run – Can execute the code
  • Knows how to start
  • Can provide input
  • Understands the output
Level 2: Trace – Can follow execution
  • Follows variable values
  • Understands control flow
  • Can predict output
Level 3: Explain – Can describe purpose
  • Understands why code exists
  • Can summarize function
  • Recognizes design patterns
Level 4: Write – Can create code
  • Can implement solutions
  • Can modify existing code
  • Can extend functionality

Exercises

CL-1: Procedural Reading

Read code step by step and trace variable values.

Learning Objectives:
  • Follow program flow
  • Trace variable values
  • Predict output
trace_example.py 
def process_items(items):
    result = []
    for item in items:
        if len(item) > 3:
            result.append(item.upper())
        else:
            result.append(item)
    return result

data = ["cat", "elephant", "dog", "butterfly"]
output = process_items(data)
print(output)

Task: Trace through this code step by step. For each iteration, write down: current item, condition result, what gets appended. Predict the output before running.

Reflection:
  • Did your prediction match the actual output?
  • Where did you have to slow down and think?
  • What would you change to improve readability?

CL-2: Functional Reading

Understand code purpose without tracing every line.

Learning Objectives:
  • Identify code purpose from structure
  • Use naming as documentation
  • Skim before deep reading
functional_example.py 
def validate_museum_record(record):
    """Check if a museum record has all required fields."""
    required_fields = ['inventory_number', 'title', 'date_acquired']

    missing = [f for f in required_fields if f not in record]

    if missing:
        return {'valid': False, 'missing': missing}

    if not record['inventory_number'].startswith('INV-'):
        return {'valid': False, 'error': 'Invalid inventory format'}

    return {'valid': True}

Task: Without tracing line-by-line: What does this function do? What are the possible return values? How would you use it?

Reflection:
  • Which elements helped you understand quickly?
  • What questions remain unanswered?
  • How would you test this function?

Deep Dive

Reading vs. Writing

Research shows that reading and writing code are distinct skills:

  • You can learn to read code without being able to write it
  • Professional developers spend 70% of their time reading code
  • Understanding generated code is sufficient for many AI-assisted tasks

Strategies for LLM-Generated Code

Skim First
Look at function names, structure, imports before details
Ask for Explanation
Have the LLM explain its own code
Test Before Understanding
Run with sample data – does it work?
Compare Versions
Ask for alternative implementations

Delocalized Plans

In real codebases, functionality is often spread across files:

  • Configuration in one file
  • Logic in another
  • Error handling in a third
  • Tests elsewhere

Understanding requires mentally assembling these pieces – a skill that transfers to reading LLM output across multiple files.
