CT

Computational Thinking

Thinking like a programmer

Computational Thinking is the ability to formulate problems in a way that a computer (or an LLM) can solve them. This doesn't require programming knowledge, but rather a specific way of structuring and analyzing problems.

Theory Exercises Deep Dive Resources

Theory

Key Points

  • Decomposition: Break complex problems into smaller, manageable parts
  • Pattern Recognition: Identify recurring structures in the problem
  • Abstraction: Distill to the essential, ignore irrelevant details
  • Algorithm Design: Develop step-by-step solutions
  • Evaluation: Critically assess solutions and improve them iteratively

Concepts

Decomposition
Breaking down complex problems into smaller, independent sub-problems
Abstraction
Focusing on relevant details while ignoring unnecessary complexity
Pattern Recognition
Identifying similarities and recurring structures across problems
Algorithm
A precise, step-by-step instruction to solve a specific problem
Pseudocode
Informal description of an algorithm in natural language

From Problem to Solution

Example Problem: "Analyze metadata from 500 museum objects"
  • Decomposition: Import → Validation → Analysis → Export
  • Abstraction: What data per object? What kind of analysis?
  • Pattern: Are there recurring data formats?
  • Algorithm: Step-by-step processing

Exercises

CT-1: Problem Decomposition

Decompose a complex task into manageable parts.

Learning Objectives:
  • Apply decomposition technique
  • Identify dependencies between sub-problems
  • Consider solution order
decomposition_example.txt 
# Task: "Create a website for a photo exhibition"

## Decomposition:
1. Content: Which photos? Descriptions? Arrangement?
2. Structure: How many pages? Navigation?
3. Design: Colors, Fonts, Layout?
4. Technology: Static HTML or CMS?
5. Hosting: Where to publish?

## Dependencies:
- Content → Structure (number of images determines pages)
- Design → Technology (some designs need JS)
- Technology → Hosting (CMS needs database)

Task: Take a current project from your work and decompose it into 5-7 sub-problems. Mark dependencies between the parts.

Reflection:
  • Which sub-problem is the most complex?
  • Which parts can be processed in parallel?
  • Where would AI-supported assistance be most helpful?

CT-2: Pseudocode Development

Describe a solution in natural language before writing code.

Learning Objectives:
  • Write pseudocode
  • Bridge the gap between idea and implementation
  • Use pseudocode as prompt
pseudocode.txt 
# Task: Detect duplicate images in a folder

## Pseudocode:
1. Collect all image files in the folder
2. For each image:
   a. Calculate a hash value (fingerprint)
   b. Store in list: [Filename, Hash]
3. Sort list by hash
4. Find adjacent entries with same hash
5. Output duplicates as pairs

Task: Use this pseudocode as a prompt for an LLM. Compare the result with an unstructured prompt like "Write a script that finds duplicate images."

Reflection:
  • How did the outputs differ?
  • Did the LLM suggest improvements to the pseudocode?
  • Where was the pseudocode too vague?

Deep Dive

Computational Thinking vs. Programming

Computational Thinking is the thinking, programming is the implementation. You can have CT without programming, but good programming needs CT.

  • CT: "How do I organize this data logically?"
  • Programming: "What data structure do I use in Python?"

CT in AI-Assisted Coding

When using LLMs for coding, CT skills become even more important:

  • Decomposition helps formulate clear, focused prompts
  • Abstraction helps provide appropriate context
  • Pattern Recognition helps identify reusable solutions
  • Algorithm thinking helps evaluate generated code

The Four Dimensions

Decomposition
Most directly transferable to prompt engineering – breaking tasks into steps
Abstraction
Critical for context engineering – what to include, what to omit
Pattern Recognition
Helps in code literacy – understanding code structure
Algorithm Design
Foundation for reviewing and correcting LLM output
