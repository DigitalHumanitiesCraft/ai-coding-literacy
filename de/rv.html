Review

Ergebnisse systematisch prüfen

Review bezeichnet die systematische Prüfung von generiertem Code gegen die definierten Anforderungen. LLM-generierter Code sieht oft korrekt aus, enthält aber systematisch Fehler. EvalPlus zeigt: Pass-Raten sinken um 19-28.9% bei gründlicherem Testing. Vertrauen ist gut, Prüfen ist besser.

Theorie Übungen Vertiefung Ressourcen

Theorie

Kernpunkte

  • EvalPlus: Pass-Raten sinken um 19-28.9% bei gründlicherem Testing
  • Tambon et al.: 10 Bug-Patterns in LLM-generiertem Code
  • 29.5% Python-Snippets von Copilot haben Sicherheitslücken (Fu et al.)
  • 5 Halluzinationskategorien: Intent Conflicting, Context Inconsistency, Dead Code, Knowledge Conflicting
  • Metamorphic Testing: Paraphrasierte Prompts zur Konsistenzprüfung
  • Systematisch testen: Normalfall, Grenzfall, Fehlerfall

Konzepte

Halluzinierte Objekte
Nicht existierende Funktionen oder Bibliotheken, die das LLM erfindet
Intent Conflicting
Code widerspricht der Anforderung – macht etwas anderes als gewünscht
Context Inconsistency
Code widerspricht dem bereitgestellten Kontext (Beispieldaten, Dokumentation)
Metamorphic Testing
Paraphrasierte Prompts zur Konsistenzprüfung – erkennt 75% fehlerhafter Programme
Traceback
Die Fehlerspur, die zeigt, wo der Fehler aufgetreten ist – von unten nach oben lesen
Logischer Fehler
Code läuft, aber das Ergebnis ist falsch – am schwierigsten zu finden

Die 5 Halluzinationskategorien

1. Intent Conflicting

Der Code macht etwas anderes als gefordert.

  • Gefordert: "Sortiere absteigend" → Code sortiert aufsteigend
  • Gefordert: "Nur .jpg" → Code verarbeitet alle Bildformate
2. Context Inconsistency

Der Code ignoriert den bereitgestellten Kontext.

  • Beispieldaten haben Spalte "ID" → Code erwartet "id"
  • Dokumentation sagt UTF-8 → Code verwendet Latin-1
3. Dead Code

Code, der nie ausgeführt wird.

  • Funktionen die definiert aber nie aufgerufen werden
  • Bedingungen die immer wahr/falsch sind
4. Knowledge Conflicting

Widerspruch zu allgemeinem Wissen.

  • Falsche API-Signaturen
  • Veraltete Syntax
  • Erfundene Bibliotheken
5. Halluzinierte Objekte

Komplett erfundene Dinge.

  • `import fictional_library`
  • `pandas.magic_transform()` – existiert nicht

Übungen

RV-1: Fehlermeldungen lesen

Tracebacks verstehen und die Ursache lokalisieren.

Lernziele:
  • Tracebacks von unten nach oben lesen
  • Die relevante Zeile identifizieren
  • Fehlertypen unterscheiden
traceback.txt 
Traceback (most recent call last):
  File "script.py", line 15, in 
    result = process_data(data)
  File "script.py", line 8, in process_data
    return data["name"].upper()
KeyError: 'name'

# Interpretation:
# 1. Letzte Zeile: Der Fehler (KeyError: 'name')
# 2. Zeile davor: Wo es passierte (Zeile 8, data["name"])
# 3. Das Dictionary 'data' hat keinen Key 'name'
# 4. Lösung: Prüfen welche Keys tatsächlich existieren

Aufgabe: Lassen Sie vom LLM Code generieren und provozieren Sie absichtlich einen Fehler (z.B. falsche Datei). Lesen Sie den Traceback und lokalisieren Sie das Problem.

Reflexion:
  • Welche Information steht in der letzten Zeile?
  • Wie finden Sie die fehlerhafte Zeile in Ihrem Code?
  • War der Fehler im generierten Code oder in Ihrer Verwendung?

RV-2: Systematische Code-Prüfung

Eine Checkliste für LLM-generierten Code durcharbeiten.

Lernziele:
  • Systematisch statt intuitiv prüfen
  • Typische LLM-Fehler erkennen
  • Eigene Checklisten entwickeln
review_checkliste.md 
# Review-Checkliste für LLM-Code

## 1. Läuft es?
- [ ] Code startet ohne Syntaxfehler
- [ ] Alle Imports sind verfügbar
- [ ] Keine erfundenen Bibliotheken

## 2. Macht es das Richtige?
- [ ] Ausgabe entspricht Anforderung
- [ ] Normalfall funktioniert
- [ ] Grenzfälle getestet (leere Eingabe, große Daten)

## 3. Macht es nichts Falsches?
- [ ] Keine unerwarteten Nebeneffekte
- [ ] Originaldaten werden nicht überschrieben
- [ ] Keine Sicherheitslücken (Paths, Injections)

## 4. Ist es robust?
- [ ] Fehlerbehandlung vorhanden
- [ ] Fehlermeldungen sind hilfreich
- [ ] Keine hardcodierten Pfade/Werte

Aufgabe: Nehmen Sie Code aus einer vorherigen Übung und arbeiten Sie die Checkliste durch. Dokumentieren Sie jeden Punkt.

Reflexion:
  • Welche Probleme hätten Sie ohne Checkliste übersehen?
  • Wie würden Sie die Checkliste für Ihre Projekte anpassen?

Vertiefung

Die drei Testfälle

Jeder Code sollte mindestens mit diesen drei Szenarien getestet werden:

Normalfall
Typische, erwartete Eingabe. "Das sollte funktionieren."
Grenzfall
Extreme Werte: leere Liste, sehr große Datei, Sonderzeichen, Umlaute.
Fehlerfall
Ungültige Eingabe: Datei existiert nicht, falsches Format, fehlende Rechte.

Metamorphic Testing

Eine einfache Technik um LLM-Konsistenz zu prüfen:

  1. Stellen Sie dieselbe Anfrage mit anderen Worten
  2. Vergleichen Sie die Ergebnisse
  3. Bei unterschiedlichen Ergebnissen: mindestens eines ist falsch

Beispiel:

  • Prompt A: "Sortiere diese Liste absteigend"
  • Prompt B: "Ordne diese Werte vom größten zum kleinsten"
  • Wenn unterschiedliche Ergebnisse → Nachprüfen!

10 häufige Bug-Patterns (nach Tambon et al.)

  1. Off-by-one – Schleife läuft einmal zu oft/zu wenig
  2. Missing Null Check – Keine Prüfung auf leere Werte
  3. Wrong Operator – `=` statt `==`, `>` statt `>=`
  4. Type Mismatch – String statt Integer, etc.
  5. Hardcoded Values – Feste Pfade, die nur lokal funktionieren
  6. Missing Error Handling – Keine try/except
  7. Resource Leak – Dateien nicht geschlossen
  8. Race Condition – Bei paralleler Verarbeitung
  9. Incorrect API Usage – Falsche Parameter, veraltete Syntax
  10. Security Vulnerabilities – SQL Injection, Path Traversal
