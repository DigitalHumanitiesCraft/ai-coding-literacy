Requirement Engineering

Anforderungen präzise formulieren

Requirement Engineering bezeichnet die systematische Formulierung von Anforderungen. Für die Arbeit mit LLMs ist das entscheidend, weil ein Prompt funktional ein Requirements-Dokument ist. Wer präzise Anforderungen formulieren kann, schreibt bessere Prompts.

Theorie Übungen Vertiefung Ressourcen

Theorie

Kernpunkte

  • Ein Prompt ist funktional ein Requirements-Dokument
  • User Stories sind niedrigschwelliger als formale Spezifikationen
  • LLMs raten bei vagen Anforderungen – sie füllen Lücken mit Annahmen
  • Akzeptanzkriterien definieren, bevor Code generiert wird
  • Grenzfälle und Ausnahmen antizipieren
  • Das LLM kann nur liefern, was Sie spezifizieren

Konzepte

User Story
Leichtgewichtiges Anforderungsformat: "Als [Rolle] möchte ich [Funktion], damit [Nutzen]"
INVEST-Kriterien
Gute Anforderungen sind: Independent, Negotiable, Valuable, Estimable, Small, Testable
Kernfragen
Was tun? Was nicht? Eingabe? Ausgabe? Woran erkenne ich Erfolg?
Funktionale Anforderung
Was das System tun soll (Eingabe → Verarbeitung → Ausgabe)
Akzeptanzkriterium
Messbarer Maßstab, ob eine Anforderung erfüllt ist
Grenzfall
Extreme oder ungewöhnliche Eingaben, die getestet werden müssen

Die fünf Kernfragen

Bevor Sie ein LLM ansprechen, beantworten Sie diese Fragen:

  1. Was soll es tun? Die Hauptfunktion in einem Satz.
  2. Was soll es nicht tun? Explizite Abgrenzungen.
  3. Was ist die Eingabe? Format, Quelle, Beispiele.
  4. Was ist die Ausgabe? Format, Ziel, Beispiele.
  5. Woran erkenne ich Erfolg? Messbare Kriterien.

User Stories für Prompts

Das User-Story-Format aus der Softwareentwicklung eignet sich hervorragend für Prompts:

user_story.txt 
"Als [WER bin ich / meine Rolle]
möchte ich [WAS soll passieren]
damit [WARUM / welchen Nutzen habe ich]"

Beispiel:
"Als Historikerin möchte ich
eine CSV-Datei mit Archivdaten in eine formatierte Bibliografie umwandeln,
damit ich sie direkt in meine Publikation übernehmen kann."

Übungen

RE-1: User Stories schreiben

Anforderungen im User-Story-Format formulieren, bevor Sie prompten.

Lernziele:
  • Das User-Story-Format anwenden
  • Akzeptanzkriterien definieren
  • Abgrenzungen explizit machen

Szenario: Sie möchten ein Tool, das Ihre Bibliografie verwaltet.

anforderung.txt 
# Vage Anforderung (schlecht):
"Ich brauche was, um meine Bibliografie zu verwalten"

# User Story (besser):
"Als Forschende möchte ich eine CSV-Datei mit
Bibliografiedaten in eine formatierte HTML-Seite
umwandeln, damit ich meine Publikationsliste
auf meiner Website anzeigen kann."

# Präzise Spezifikation (am besten):
# Eingabe: CSV mit Spalten (Autor, Titel, Jahr, Zeitschrift)
# Ausgabe: HTML mit alphabetisch sortierter Liste
# Format: Autor (Jahr): Titel. In: Zeitschrift.
# Nicht im Scope: PDF-Verlinkung, Dubletten-Erkennung

Aufgabe: Schreiben Sie für Ihr eigenes Projekt: 1) Eine User Story, 2) Eingabe und Ausgabe, 3) Drei Akzeptanzkriterien, 4) Zwei Abgrenzungen (Was soll es nicht tun?)

Reflexion:
  • Welche Anforderungen waren implizit (in Ihrem Kopf)?
  • Was hätte das LLM ohne diese Spezifikation geraten?
  • Wie schwer war es, Abgrenzungen zu formulieren?

RE-2: Akzeptanzkriterien testen

Code systematisch gegen definierte Kriterien prüfen.

Lernziele:
  • Testfälle aus Anforderungen ableiten
  • Systematisch Grenzfälle durchspielen
  • Abweichungen dokumentieren

Aufgabe: Nehmen Sie den generierten Code aus RE-1 und testen Sie ihn gegen Ihre Akzeptanzkriterien. Erstellen Sie eine Checkliste und prüfen Sie jeden Punkt.

Reflexion:
  • Wie viele Ihrer Kriterien wurden erfüllt?
  • Welche Fehler hätten Sie ohne Kriterien übersehen?
  • Was müssen Sie im Prompt anpassen?

Vertiefung

INVEST-Kriterien für Prompts

Die INVEST-Kriterien helfen, gute Anforderungen zu schreiben:

Independent (Unabhängig)
Eine Anfrage pro Prompt. Nicht: "Mach A und dann B und C".
Negotiable (Verhandelbar)
Details können angepasst werden. Das LLM kann Rückfragen stellen.
Valuable (Wertvoll)
Jede Anforderung liefert konkreten Nutzen.
Estimable (Schätzbar)
Sie können einschätzen, ob es in einem Prompt lösbar ist.
Small (Klein)
Eine überschaubare Aufgabe. Komplexes in Teile zerlegen.
Testable (Testbar)
Sie wissen vorher, wie Sie prüfen, ob es funktioniert.

Typische Fehler bei Anforderungen

  • "Es soll gut aussehen" – Was bedeutet "gut"? Konkretisieren!
  • "Wie bei X" – Das LLM kennt X vielleicht nicht oder anders.
  • Implizite Annahmen – "Natürlich soll es auch Y können" – nein, nicht natürlich.
  • Keine Grenzfälle – Was bei leerer Eingabe? Bei Sonderzeichen?
